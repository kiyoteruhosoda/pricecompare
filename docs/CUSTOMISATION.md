# Customising the template for a new app

This template is meant to be forked. Values that identify **your** app
(name, package, colours, icons) live in a small number of places.
This guide lists every one of them.

---

## TL;DR — 6-step fork

1. **Rename Dart + Android package** — `scripts/rename_app.sh my_cool_app com.mycompany.coolapp`  ([§5](#5-package-import-cascade))
2. **Edit Dart-side identity** — `lib/shared/config/app_config.dart`  ([§2](#2-dart-side-appconfig))
3. **Edit pubspec metadata** — `description:`, `version:`  ([§3](#3-pubspecyaml))
4. **Edit Android label** — `android/app/src/main/AndroidManifest.xml`  ([§4](#4-android-native-identifiers))
5. **Swap launcher icons** — drop new PNGs into `assets/icon/`, run `dart run flutter_launcher_icons`  ([§6](#6-launcher-icons))
6. **Update README.md line 1 + verify** — `flutter clean && flutter pub get && dart analyze && flutter test`  ([§8](#8-readmemd), [§11](#11-verification))

iOS setup is out of scope — see [§9](#9-ios--not-yet-configured).

---

## 2. Dart-side (`AppConfig`)

`lib/shared/config/app_config.dart` is the single source of truth for every
per-app identity value that can live in Dart.

| Field | Effect |
|---|---|
| `appName` | `MaterialApp` title, drawer header, About header, License page header |
| `appDescription` | About page subtitle |
| `appTagline` | Drawer subtitle |
| `homeSubtitle` | Home "Welcome" subtitle |
| `homeCardTitle` | Home first card title |
| `designSystemLabel` | About page "Design System" row |
| `designSystemName` | Registered license entry title |
| `designSystemLicense` | Registered license entry body |
| `designSystemUrl` | Registered license entry body |

Brand colour is **not** here — it lives in `AppColors.brand`
(`lib/shared/theme/app_colors.dart`). Change it there; `AppTheme.light` /
`AppTheme.dark` read it automatically.

Version number and build metadata are **not** here either — they come from
the auto-generated `lib/shared/build_info.dart` (see
`scripts/generate_build_info.sh`).

## 3. `pubspec.yaml`

| Line | Field | Notes |
|---|---|---|
| 1 | `name:` | Dart package name. `scripts/rename_app.sh` rewrites this. |
| 2 | `description:` | One-line description shown on pub.dev / `flutter pub`. Edit by hand. |
| 4 | `version:` | Semver + build number. Bumped by `scripts/bump_version.sh`. |
| — | `flutter_launcher_icons:` | Adaptive icon config — see [§6](#6-launcher-icons). |

## 4. Android native identifiers

| File | Field | Rewritten by `rename_app.sh`? |
|---|---|---|
| `android/app/build.gradle` | `namespace` | yes |
| `android/app/build.gradle` | `applicationId` | yes |
| `android/app/src/main/AndroidManifest.xml` | `android:label` | **no — edit by hand** |
| `android/app/src/main/kotlin/.../MainActivity.kt` | `package` declaration + directory path | yes |

`AndroidManifest.xml`'s `android:label` is the name the OS shows under the
launcher icon. `AppConfig.appName` does **not** propagate here — Android
reads the label from the manifest at install time.

## 5. Package-import cascade

Flutter's `package:<name>/...` scheme means the Dart package name appears in
every absolute import in the codebase (currently ~150 files across
`lib/`, `test/`, `integration_test/`). Renaming is a mechanical rewrite.

Use the shipped script:

```bash
scripts/rename_app.sh my_cool_app com.mycompany.coolapp
```

It performs a pre-flight check (refuses to run on a dirty tree), rewrites
all `package:flutterbase/...` imports, updates `pubspec.yaml`'s `name:`,
rewrites the Android `namespace` and `applicationId`, moves the Kotlin
source tree to the new package directory, and rewrites `MainActivity.kt`'s
`package` line.

The script deliberately does **not** touch `AppConfig`, the pubspec
`description`/`version`, the Android manifest label, or launcher icons —
those are human judgement calls. The script prints a "next steps"
checklist when it finishes.

Requires GNU sed. On macOS, install via `brew install gnu-sed` and invoke
as `SED=gsed scripts/rename_app.sh ...`.

Post-run sanity check:

```bash
grep -r "package:flutterbase" lib test integration_test  # should be empty
```

## 6. Launcher icons

Launcher icons are generated by
[`flutter_launcher_icons`](https://pub.dev/packages/flutter_launcher_icons)
from two source PNGs checked into the repo:

- `assets/icon/app_icon.png` — 1024×1024, square, used for the legacy
  square icon.
- `assets/icon/app_icon_foreground.png` — 1024×1024, transparent
  background, with the foreground art contained inside the central ~66%
  safe zone. Used for the Android 8+ adaptive icon foreground layer.

Workflow:

1. Replace both PNGs in `assets/icon/` with your own (keep the same
   filenames and dimensions).
2. Open `pubspec.yaml` and update
   `flutter_launcher_icons.adaptive_icon_background` to your brand hex
   colour (e.g. `"#264AF4"`). This is the colour that shows behind your
   foreground art when the OS masks the icon into a circle/squircle/etc.
3. Update the matching value in
   `android/app/src/main/res/values/colors.xml`
   (`<color name="ic_launcher_background">…</color>`). This same colour
   is reused by the Android 12+ splash screen
   (`values-v31/styles.xml`, attribute
   `android:windowSplashScreenBackground`) and by the adaptive-icon
   background layer, so both native surfaces match the Flutter brand
   colour. Without this, Android 12+ launchers and the cold-start splash
   render the icon on a white plate.
4. Run the generator:

   ```bash
   dart run flutter_launcher_icons
   ```

5. Review the output under
   `android/app/src/main/res/mipmap-*/ic_launcher.png` and
   `mipmap-anydpi-v26/ic_launcher.xml`. Commit the regenerated files —
   they are checked in intentionally so devs don't need to run the
   generator on every clone.

For Android 13+ "themed icons", the generator also writes a monochrome
layer driven by `flutter_launcher_icons.adaptive_icon_monochrome` in
`pubspec.yaml`. It defaults to the foreground PNG, which works for most
logos; supply a dedicated black-on-transparent silhouette if yours
doesn't theme well.

Generation is a **fork-time** (and icon-change-time) action, not a
per-build action. Do not wire it into CI.

## 9. iOS — not yet configured

This template does not ship with an `ios/` directory. When you are ready
to support iOS:

1. `flutter create --platforms=ios .`
2. Edit `ios/Runner/Info.plist`:
   - `CFBundleDisplayName` — shown under the launcher icon
   - `CFBundleIdentifier` — reverse-DNS app identifier (match your Android `applicationId` unless you have a reason not to)
3. Swap the icons in `ios/Runner/Assets.xcassets/AppIcon.appiconset/`
   (or let the generator handle them — see next step).
4. Re-enable the `ios:` section in the `flutter_launcher_icons:` block of
   `pubspec.yaml`:

   ```yaml
   flutter_launcher_icons:
     android: "ic_launcher"
     ios: true
     remove_alpha_ios: true
     image_path: "assets/icon/app_icon.png"
     adaptive_icon_background: "#264AF4"
     adaptive_icon_foreground: "assets/icon/app_icon_foreground.png"
     min_sdk_android: 21
   ```

5. Re-run `dart run flutter_launcher_icons`.

`AppConfig` does **not** propagate to iOS — iOS reads its display name and
bundle ID from `Info.plist`, which is the platform's source of truth.

## 9b. Android signing — shared debug keystore

Debug builds are signed with `android/app/debug.keystore` if it exists;
otherwise the Android SDK falls back to `~/.android/debug.keystore`, which
is **per-machine**.  That fallback causes
`INSTALL_FAILED_UPDATE_INCOMPATIBLE` when a device has an APK from one
machine (or CI runner) installed and you push a build from a different
machine — Android refuses to replace an APK whose signature differs from
the new one.

One-time setup per fork:

```bash
bash scripts/generate_debug_keystore.sh
git add android/app/debug.keystore
git commit -m "chore(android): pin shared debug keystore"
```

The password (`android`) and alias (`androiddebugkey`) are Android SDK
defaults and are **not secrets** — a debug keystore only signs debug
APKs, which are never distributed.  Do not reuse these values for
release signing (§9c).

## 9c. Android APK / AAB output filenames

After every `flutter build apk` / `flutter build appbundle`,
`android/app/build.gradle` writes a **per-app named copy** alongside
Gradle's default output.  Base-name resolution:

1. Gradle property `app.archivesBaseName` (set in
   `android/gradle.properties` or passed via `-P`).
2. Last segment of `applicationId` — e.g. `com.mycompany.coolapp` →
   `coolapp`.  `rename_app.sh` rewrites `applicationId`, so forks get a
   per-app name automatically without further edits.

Why copies and not renames? Flutter's CLI post-build step hard-codes
the default filenames (`app-release.aab`, `app-release.apk`).  Renaming
the originals via `base.archivesName` or
`applicationVariants.outputFileName` breaks `flutter build` with
"Gradle build failed to produce an .aab file."  Leaving the defaults in
place keeps Flutter CLI happy, and the named copy lives next to it.

Produced artifacts (both files exist after every build):

| Kind | Path |
|---|---|
| Default APK (used by Flutter CLI) | `build/app/outputs/flutter-apk/app-<buildType>.apk` |
| Per-app APK copy | `build/app/outputs/flutter-apk/<base>-<versionName>-<buildType>.apk` |
| Default AAB (used by Flutter CLI) | `build/app/outputs/bundle/<buildType>/app-<buildType>.aab` |
| Per-app AAB copy | `build/app/outputs/bundle/<buildType>/<base>-<versionName>-<buildType>.aab` |

To override the base name explicitly, add to `android/gradle.properties`:

```properties
app.archivesBaseName=my_cool_app
```

For CI uploads, point `actions/upload-artifact` at the per-app glob
(e.g. `build/app/outputs/bundle/release/${APP_NAME}-*.aab`) so the
downloaded artifact already carries the per-app filename.

## 10. Explicitly out of scope

None of the following are touched by this template, by `rename_app.sh`,
or by anything in this guide. Set them up per your own deployment needs:

- Native splash screen (`flutter_native_splash`) — the template relies
  solely on the Android 12+ system splash (see §6); no Dart-rendered
  splash is shown.
- Build flavours / product flavors.
- Firebase / Crashlytics / Analytics / `google-services.json`.
- Signing keys, keystore generation, Play Console upload.
- Store listing metadata (screenshots, long description, etc.).
- `lib/shared/build_info.dart` — auto-generated by
  `scripts/generate_build_info.sh`; do not edit by hand.

## 11. Verification

After forking + customising, run these:

```bash
flutter clean
flutter pub get
dart run flutter_launcher_icons          # if you changed icons
dart analyze --fatal-infos --fatal-warnings
flutter test
```

Then do a visual smoke pass:

| Surface | What to check |
|---|---|
| Drawer header | Shows `AppConfig.appName` and `AppConfig.appTagline`. |
| Home tab | "Welcome" title, `AppConfig.homeSubtitle`, first card uses `AppConfig.homeCardTitle`. |
| About page | Header shows `AppConfig.appName` + `AppConfig.appDescription`, version row matches pubspec. |
| License page | Opens via "Licenses" menu item, shows the DADS entry with your `designSystemName` / `designSystemLicense` / `designSystemUrl`. |
| Launcher icon | Match between `assets/icon/app_icon.png` and the icon shown on the device. |

Grep gates to catch identity drift:

```bash
grep -rn "'FlutterBase'"       lib/ test/   # should return only AppConfig
grep -rn "'DADS Design System'" lib/ test/  # should return only AppConfig
```
